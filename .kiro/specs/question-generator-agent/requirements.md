# Requirements Document

## Introduction

The Question Generator Agent is a Python function (`generate_questions`) in `agents/question_generator.py` that takes a researcher output dictionary and an API key as input, then uses Gemini 2.0 Flash to generate exactly 10 company-specific interview questions in a single LLM call. It compresses the research data before passing it to the LLM, validates the output structure and distribution, retries once if the question count is incorrect, and saves all questions to SQLite via `save_question()`. The agent follows the `safe_llm_call` template pattern and raises `QuestionGenerationError` when generation fails after retry.

## Glossary

- **Question_Generator_Agent**: The Python function `generate_questions` in `agents/question_generator.py` that generates 10 interview questions from researcher output via a single Gemini LLM call
- **Research_Data**: The validated Python dictionary with exactly 8 keys returned by the Researcher_Agent, containing company-specific interview intelligence
- **Question_Dict**: A dictionary representing a single generated question with exactly 7 keys: `id`, `category`, `question`, `ideal_keywords`, `difficulty`, `follow_ups`, and `scoring_hint`
- **Question_Distribution**: The required category allocation for the 10 questions: 4 technical, 3 behavioral, 2 situational, 1 curveball
- **QuestionGenerationError**: A custom exception raised when question generation fails after exhausting all retry attempts
- **Safe_LLM_Call**: A wrapper function following the steering template that handles retries, markdown stripping, JSON parsing, and error logging for all LLM calls
- **MAX_TOKENS_COMPLEX**: The constant value 1000 from `core/config.py` used as the maximum output token limit for the question generator LLM call
- **RATE_LIMIT_SLEEP**: The constant value 4 from `core/config.py` representing the mandatory seconds to wait between consecutive LLM calls
- **TOTAL_QUESTIONS**: The constant value 10 from `core/config.py` representing the exact number of questions to generate
- **MIN_QUESTION_LENGTH**: The constant value 20 from `core/config.py` representing the minimum character length for valid question text
- **FOLLOW_UP_COUNT**: The constant value 2 from `core/config.py` representing the exact number of follow-up questions per question
- **Data_Compression**: The process of serializing Research_Data using `json.dumps(data, separators=(',',':'))` to minimize token usage in the LLM prompt

## Requirements

### Requirement 1: Generate Interview Questions from Research Data

**User Story:** As a job seeker, I want the system to generate 10 company-specific interview questions based on research findings, so that I can practice with realistic, targeted questions.

#### Acceptance Criteria

1. WHEN the Question_Generator_Agent receives a valid Research_Data dictionary (containing all 8 required researcher keys: company, role, interview_rounds, key_topics, difficulty, culture_keywords, known_question_types, red_flags_to_test with non-empty values) and a valid api_key string (non-empty, non-whitespace), THE Question_Generator_Agent SHALL generate exactly `TOTAL_QUESTIONS` interview questions in a single LLM call and return a list of `TOTAL_QUESTIONS` Question_Dict objects
2. THE Question_Generator_Agent SHALL compress Research_Data using `json.dumps(research_data, separators=(',',':'))` before including it in the LLM prompt to minimize token usage
3. THE Question_Generator_Agent SHALL make exactly one LLM call per successful invocation using the Safe_LLM_Call pattern with `MAX_TOKENS_COMPLEX` as the token limit
4. IF another LLM call preceded the current invocation in the same session, THEN THE Question_Generator_Agent SHALL call `time.sleep(RATE_LIMIT_SLEEP)` before making the LLM call
5. THE Question_Generator_Agent SHALL return a list of Question_Dict objects where each dict contains exactly these 7 keys: `id` (str, UUID4 format generated by the agent using `uuid.uuid4()`), `category` (str, one of "technical", "behavioral", "situational", "curveball"), `question` (str, minimum `MIN_QUESTION_LENGTH` characters), `ideal_keywords` (list of at least 3 strings), `difficulty` (int, 1-10), `follow_ups` (list of exactly `FOLLOW_UP_COUNT` strings), `scoring_hint` (str)
6. IF the Research_Data dictionary is missing any of the 8 required keys or the api_key is empty or whitespace-only, THEN THE Question_Generator_Agent SHALL raise QuestionGenerationError with a message indicating which input parameter failed validation

### Requirement 2: Enforce Question Category Distribution

**User Story:** As a job seeker, I want a balanced mix of question types, so that I can practice across all interview dimensions.

#### Acceptance Criteria

1. THE Question_Generator_Agent SHALL produce questions with exactly this category distribution: 4 questions with category "technical", 3 questions with category "behavioral", 2 questions with category "situational", and 1 question with category "curveball"
2. WHEN the LLM returns questions where the category counts do not match the required distribution of 4 technical, 3 behavioral, 2 situational, and 1 curveball, THE Question_Generator_Agent SHALL retry the LLM call exactly once (for a maximum of 2 total attempts) to obtain the correct distribution
3. WHEN the Question_Generator_Agent receives the LLM response, THE Question_Generator_Agent SHALL validate that each question has a category field matching one of the four valid values ("technical", "behavioral", "situational", "curveball") as an exact case-sensitive string, and that the category counts match the required distribution
4. IF the category distribution validation fails after both attempts (initial call and retry), THEN THE Question_Generator_Agent SHALL raise QuestionGenerationError with a message indicating the expected versus actual distribution
5. IF any question contains a category value that does not exactly match one of "technical", "behavioral", "situational", or "curveball", THEN THE Question_Generator_Agent SHALL treat the response as a distribution validation failure

### Requirement 3: Enforce Progressive Difficulty

**User Story:** As a job seeker, I want questions to get progressively harder, so that I can warm up with easier questions before facing difficult ones.

#### Acceptance Criteria

1. THE Question_Generator_Agent SHALL instruct the LLM to order questions from easiest (question 1) to hardest (question 10) within the system prompt, specifying that difficulty must increase monotonically with question position
2. THE Question_Generator_Agent SHALL assign a difficulty value (integer 1 to 10) to each question where question N has difficulty N, representing progressive difficulty from Q1 (easiest) to Q10 (hardest)
3. WHEN the LLM returns questions, THE Question_Generator_Agent SHALL validate that exactly 10 questions are present and that each question's difficulty field is an integer between 1 and 10 inclusive
4. IF any question's difficulty field is not an integer between 1 and 10 or the difficulty values are not strictly ascending from question 1 to question 10, THEN THE Question_Generator_Agent SHALL correct the difficulty values by assigning sequential integers 1 through 10 based on question position in the list

### Requirement 4: Validate Question Output Structure

**User Story:** As a developer, I want generated questions validated before use, so that downstream agents receive correctly structured data.

#### Acceptance Criteria

1. WHEN the LLM returns a response, THE Question_Generator_Agent SHALL validate that the response contains a "questions" key holding a list of exactly `TOTAL_QUESTIONS` dictionaries
2. THE Question_Generator_Agent SHALL validate that each Question_Dict contains all 7 required keys: `id`, `category`, `question`, `ideal_keywords`, `difficulty`, `follow_ups`, and `scoring_hint`
3. THE Question_Generator_Agent SHALL validate that `id` is a non-empty string, `category` is one of "technical", "behavioral", "situational", or "curveball", `question` is a string of at least `MIN_QUESTION_LENGTH` characters, `ideal_keywords` is a non-empty list of strings, `difficulty` is an integer between 1 and 10, `follow_ups` is a list of strings, and `scoring_hint` is a non-empty string
4. IF any Question_Dict is missing a required key or any value fails type or constraint validation, THEN THE Question_Generator_Agent SHALL reject the entire response, append a corrective JSON-only instruction to the prompt, wait `RATE_LIMIT_SLEEP` seconds, and retry the LLM call exactly once (maximum 2 total attempts)
5. IF any `question` text is fewer than `MIN_QUESTION_LENGTH` characters, THEN THE Question_Generator_Agent SHALL reject the entire response, append a corrective JSON-only instruction to the prompt, wait `RATE_LIMIT_SLEEP` seconds, and retry the LLM call exactly once (maximum 2 total attempts)
6. IF validation fails on both attempts (initial call and retry), THEN THE Question_Generator_Agent SHALL raise QuestionGenerationError with a message indicating the agent name, the validation error encountered, and that it failed after 2 attempts

### Requirement 5: Normalize Follow-Up Questions

**User Story:** As a developer, I want each question to have exactly 2 follow-ups, so that the evaluator agent can rely on a consistent structure.

#### Acceptance Criteria

1. THE Question_Generator_Agent SHALL ensure each Question_Dict contains a `follow_ups` list of exactly `FOLLOW_UP_COUNT` strings, where each string is at least `MIN_QUESTION_LENGTH` characters long
2. WHEN the LLM returns a question with fewer than `FOLLOW_UP_COUNT` follow-up items, THE Question_Generator_Agent SHALL pad the `follow_ups` list by appending category-appropriate predefined follow-up strings (e.g., "Can you provide a specific example from your experience?" for behavioral, "Can you walk me through your technical approach?" for technical) until the list contains exactly `FOLLOW_UP_COUNT` items
3. WHEN the LLM returns a question with more than `FOLLOW_UP_COUNT` follow-up items, THE Question_Generator_Agent SHALL trim the `follow_ups` list to exactly `FOLLOW_UP_COUNT` items by keeping only the first `FOLLOW_UP_COUNT` entries
4. IF any item in the `follow_ups` list is not a string or is an empty string, THEN THE Question_Generator_Agent SHALL replace that item with a category-appropriate predefined follow-up string
5. THE Question_Generator_Agent SHALL apply follow-up normalization after the initial LLM response is parsed and after structure validation passes, but before saving questions to the database

### Requirement 6: Retry Logic for Incorrect Question Count

**User Story:** As a user, I want the system to self-correct when the LLM generates the wrong number of questions, so that I always receive exactly 10 questions.

#### Acceptance Criteria

1. WHEN the parsed LLM response contains a `questions` list with a length not equal to `TOTAL_QUESTIONS`, THE Question_Generator_Agent SHALL retry exactly once by re-sending the original prompt with an appended count correction instruction stating that exactly `TOTAL_QUESTIONS` questions are required
2. WHEN the Question_Generator_Agent prepares to make the retry LLM call for question count correction, THE Question_Generator_Agent SHALL wait `RATE_LIMIT_SLEEP` seconds before executing the retry call
3. IF the retry response also contains a `questions` list with a length not equal to `TOTAL_QUESTIONS`, THEN THE Question_Generator_Agent SHALL raise QuestionGenerationError with a message indicating the expected count (`TOTAL_QUESTIONS`) and the actual count received
4. THE Question_Generator_Agent SHALL limit total LLM call attempts to a maximum of 2 (1 initial + 1 retry) for question count correction, independent from the JSON parse retry performed by safe_llm_call
5. WHEN the retry response contains a `questions` list with a length equal to `TOTAL_QUESTIONS`, THE Question_Generator_Agent SHALL return the retry response as the successful result

### Requirement 7: Persist Questions to Database

**User Story:** As a user, I want my generated questions saved to the database, so that they persist across sessions and can be reviewed later.

#### Acceptance Criteria

1. WHEN the Question_Generator_Agent successfully generates and validates `TOTAL_QUESTIONS` questions, THE Question_Generator_Agent SHALL save each question to SQLite by calling `save_question()` from `core.database` for each Question_Dict in the list, passing both the Question_Dict (all 7 keys) and the `session_id` string to associate the question with the current user session
2. THE Question_Generator_Agent SHALL save all `TOTAL_QUESTIONS` questions to the database before returning the question list to the caller
3. IF a database write fails for any question, THEN THE Question_Generator_Agent SHALL raise QuestionGenerationError indicating the question index (0-based) that failed to save and the underlying error reason, without suppressing the original exception
4. IF a database write fails after some questions have already been saved, THEN THE Question_Generator_Agent SHALL NOT attempt to roll back previously saved questions, allowing partial writes to remain in the database
5. THE Question_Generator_Agent SHALL accept `session_id` as a required parameter and pass it to each `save_question()` call alongside the complete Question_Dict

### Requirement 8: Handle Unknown Companies Gracefully

**User Story:** As a job seeker targeting a lesser-known company, I want the system to still generate useful interview questions, so that I can practice regardless of company recognition.

#### Acceptance Criteria

1. WHEN the Research_Data dictionary contains an `error_flag` key set to True, THE Question_Generator_Agent SHALL generate exactly `TOTAL_QUESTIONS` questions using the `role`, `difficulty`, and `key_topics` fields from Research_Data as the basis for question content and difficulty calibration
2. WHEN Research_Data contains an `error_flag` key set to True, THE Question_Generator_Agent system prompt SHALL instruct the LLM to generate questions based solely on the provided role, experience level, and key_topics without referencing company-specific culture, interview processes, or company name in the generated questions
3. WHEN Research_Data contains an `error_flag` key set to True, THE Question_Generator_Agent SHALL proceed with question generation without raising an error, treating the available Research_Data fields as valid input
4. WHEN Research_Data contains an `error_flag` key set to True, THE Question_Generator_Agent SHALL return output conforming to the standard Question_Dict JSON contract (a list of objects each having: id, category, question, ideal_keywords, difficulty, follow_ups, scoring_hint) identical in structure to the output produced when `error_flag` is absent or False

### Requirement 9: Error Handling and QuestionGenerationError

**User Story:** As a developer, I want clear error signaling when question generation fails, so that the orchestrator can transition to the ERROR state.

#### Acceptance Criteria

1. IF the LLM call fails after exhausting all retry attempts (2 attempts in safe_llm_call for JSON parse and 1 additional retry for question count validation), THEN THE Question_Generator_Agent SHALL raise QuestionGenerationError with a message that includes the string "Question_Generator_Agent" and a failure reason distinguishing the failure mode (JSON parse failure, wrong question count, API error)
2. IF the validated question list does not satisfy the required category distribution after 1 retry attempt, THEN THE Question_Generator_Agent SHALL raise QuestionGenerationError with a message that includes the string "Question_Generator_Agent", the expected distribution, and the actual distribution received
3. IF any question text is fewer than `MIN_QUESTION_LENGTH` characters after 1 retry attempt, THEN THE Question_Generator_Agent SHALL raise QuestionGenerationError with a message that includes the string "Question_Generator_Agent" and the zero-based index of the first question that failed the minimum length check
4. THE QuestionGenerationError class SHALL be defined in `agents/question_generator.py` as a subclass of Exception with a `message` attribute of type `str` that is set via the constructor
5. IF the LLM API call raises a network or authentication error (e.g., connection timeout, invalid API key) after 2 attempts, THEN THE Question_Generator_Agent SHALL raise QuestionGenerationError with a message that includes the string "Question_Generator_Agent" and the original exception's error text

### Requirement 10: System Prompt and Configuration Compliance

**User Story:** As a developer, I want the question generator agent to follow system prompt and configuration conventions, so that LLM responses are consistently formatted and the codebase remains maintainable.

#### Acceptance Criteria

1. THE Question_Generator_Agent system prompt SHALL end with the exact text: "Return ONLY a JSON object. No markdown. No explanation. No text before or after. Pure JSON only."
2. THE Question_Generator_Agent system prompt SHALL instruct the LLM to return a JSON object with a "questions" key containing a list of exactly `TOTAL_QUESTIONS` question objects, each with keys: "id", "category", "question", "ideal_keywords", "difficulty", "follow_ups", and "scoring_hint"
3. THE Question_Generator_Agent SHALL import `MAX_TOKENS_COMPLEX` from `core.config` and use it as the sole value for the `max_output_tokens` generation config parameter
4. THE Question_Generator_Agent SHALL import `RATE_LIMIT_SLEEP` from `core.config` and use it as the duration for all `time.sleep()` calls that enforce rate limiting between consecutive LLM calls; retry-backoff sleeps within the error-handling retry loop are excluded from this constraint
5. THE Question_Generator_Agent SHALL contain no hardcoded numeric literals for token limits, sleep durations, question counts, minimum lengths, or follow-up counts; each such value SHALL reference one of the following named constants imported from `core.config`: `MAX_TOKENS_COMPLEX`, `RATE_LIMIT_SLEEP`, `TOTAL_QUESTIONS`, `MIN_QUESTION_LENGTH`, `FOLLOW_UP_COUNT`
6. THE Question_Generator_Agent system prompt SHALL specify the exact category distribution (4 technical, 3 behavioral, 2 situational, 1 curveball) and SHALL require difficulty to progress from easy to hard across the 10 questions in sequential order
